📚 What is a SOC (Security Operations Center)?

Think of a SOC like a 24/7 security guard for computer networks. Just like how security guards monitor buildings for break-ins, our SOC monitors computer networks for cyber attacks, malware, and suspicious activities.

Our SOC platform is like a mission control center where security analysts (that's us!) watch for threats, investigate suspicious activities, and respond to security incidents. It's like having super-powered security cameras that can see digital threats instead of physical ones.

Our main job: Keep the company's digital assets safe from hackers, malware, data theft, and other cyber threats.

Integration Layer Deep Dive

Step-by-step data flow and processing pipeline

🔗 Source Connectors
  • SIEM Connectors: Splunk REST API, QRadar API
  • EDR Connectors: CrowdStrike Falcon, Microsoft Defender
  • Firewall Connectors: Palo Alto, FortiGate, pfSense
  • Cloud Security: AWS CloudTrail, Azure Sentinel
⚙️ Processing Components
  • Format Normalizer: Custom → OCSF transformation
  • Event Enricher: GeoIP, DNS, threat intel lookup
  • Correlation Engine: Pattern matching & ML scoring
  • Alert Aggregator: Deduplication & prioritization

🔄 Step-by-Step Integration Flow

1
Event Collection

Source connectors poll security systems every 30-300 seconds based on criticality

Protocols: REST API, Syslog, SNMP, File-based polling
2
Format Detection & Parsing

Intelligent parsing identifies event format (CEF, LEEF, JSON, XML) and extracts fields

Error handling: Invalid events → quarantine queue for manual review
3
OCSF Transformation

Events mapped to OCSF schema classes (Network, System, Security Finding, Authentication)

Preserves original data in 'unmapped' field for forensic analysis
4
Event Enrichment

GeoIP lookup, DNS resolution, user/asset context, threat intelligence matching

Timeout: 500ms max per enrichment to maintain real-time performance
5
ML Feature Extraction

Generate feature vectors for ML models (behavioral, statistical, contextual features)

Features: time-series patterns, frequency analysis, anomaly scores
6
Correlation & Scoring

ML models score events, correlation engine groups related events into incidents

Auto-escalation: Scores > 0.8 create incidents automatically
7
Storage & Distribution

Events stored in PostgreSQL, real-time updates broadcast via WebSocket to connected clients

Archive: Hot data 30 days, warm data 1 year, cold data 7 years

🧠 ML Feature Requirements
Required Input Features
  • • ml_timestamp - Event time in UTC epoch
  • • ml_source_category - SIEM|EDR|Firewall|Cloud
  • • ml_severity_numeric - 1-10 scale (1=low, 10=critical)
  • • ml_event_frequency - Events per hour from same source
  • • ml_user_context - User risk score (0-1)
  • • ml_asset_criticality - Asset importance (0-1)
  • • ml_network_zone - DMZ|Internal|External|Unknown
  • • ml_time_of_day - Business|After-hours|Weekend
Optional Enrichment Features
  • • ml_geolocation_risk - Geographic risk score
  • • ml_threat_intel_match - IOC match confidence
  • • ml_behavioral_anomaly - Deviation from baseline

🚨 Critical: Attribute Naming Conventions

⚠️ RESERVED PREFIXES - DO NOT USE

  • ocsf_* - Reserved for OCSF schema fields
  • raw_* - Reserved for original event data
  • sys_* - Reserved for system metadata
  • db_* - Reserved for database operations

✅ ML TEAM PREFIXES - USE THESE

  • ml_* - All ML-related fields and outputs
  • feat_* - Feature engineering outputs
  • pred_* - Model prediction results
  • conf_* - Confidence and metadata

📋 Data Type Requirements

Timestamps: Always UTC Unix epoch (seconds, not milliseconds)
Scores: Normalized 0.0-1.0 float values
Categories: Lowercase snake_case strings
Arrays: JSON arrays, max 100 elements
Null handling: Use null, never empty strings or -1
📊 Model Performance Expectations
Latency: < 100ms per prediction
Throughput: 1000+ predictions/second
Accuracy Target: > 85% on validation set
False Positive Rate: < 15%
Model Updates: Weekly retraining cycle
Fallback: Rule-based system if ML unavailable
